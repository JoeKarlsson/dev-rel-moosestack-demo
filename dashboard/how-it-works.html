MooseStack + ClickHouse ← Dashboard

How This Works

Two data layers — real-time GitHub events and historical marketing snapshots — unified in ClickHouse and served through typed TypeScript APIs. Built entirely on MooseStack.

Architecture Overview

GitHub webhook → Flask adapter :3001 → POST /ingest/GithubEvent → Redpanda stream → Transform → GithubProcessed ~300ms end-to-end
Python backfill → POST /ingest/WeeklySnapshot → Redpanda stream → WeeklySnapshot 18 weeks of history
ClickHouse tables → Materialized Views → MooseStack APIs → Dashboard polls every 10–30s

MooseStack Primitives Used

IngestPipeline
A TypeScript interface becomes an HTTP ingest endpoint, Redpanda topic, and ClickHouse table simultaneously. No config files.
stream.addTransform
Fan-out between streams with typed transforms. GitHub raw events are normalized into GithubProcessed with dead-letter queue support.
MaterializedView
Pre-aggregated ClickHouse views defined in typed SQL. GithubDailyMV buckets events by day; AiSourceWeeklyMV aggregates AI referrals by source.
Api
Consumption endpoints with typed request params and response shapes. MooseCache provides cache-aside with configurable TTLs (30s–1h).
Workflow + Task
Temporal-backed scheduled jobs. snapshotSync runs weekly (Sunday midnight UTC) to refresh the marketing snapshot from live data sources.
DeadLetterQueue
Failed transforms land in GithubDeadLetter table with the original event and error message. Observable, replayable.

The Core Insight: Type = Schema = API

In MooseStack, a TypeScript interface is the single source of truth. Define it once — the framework generates the ClickHouse table DDL, the Redpanda topic, the HTTP ingest endpoint, and validates incoming JSON against the type at the boundary.

// One interface declaration creates the entire pipeline:
export interface GithubProcessed {
  eventId: Key<string>;   // → PRIMARY KEY in ClickHouse
  timestamp: DateTime;       // → DateTime('UTC') column
  eventType: string;
  repo: string;
  actor: string;
  action: string;
  issueNumber: number;      // → Float64 in ClickHouse
  issueTitle: string;
  forkFullName: string;
}

export const GithubProcessedPipeline = new IngestPipeline<GithubProcessed>(
  "GithubProcessed",
  {
    table: { orderByFields: ["eventId", "timestamp", "eventType"] },
    stream: true,
    ingestApi: false,  // internal only — written via transform
  }
);
No YAML. No config files. No separate schema registry. Add a field to the interface and MooseStack issues the ALTER TABLE ADD COLUMN automatically on the next moose dev reload.

ClickHouse handles both simultaneously. The /api/devrel-health endpoint runs a real-time 7-day GitHub count and a historical marketing snapshot query in the same request. Columnar storage means both return in milliseconds regardless of total row count.

Why ClickHouse

Columnar storage — analytics queries scan only the columns they need. The AI referrals aggregation across 18 weeks reads 2 columns, not 25. That's why it runs in single-digit milliseconds.

Materialized Views update on insert — when a new GitHub event lands in GithubProcessed, ClickHouse automatically updates GithubDailyMetric (the star velocity pre-aggregate). The API reads the MV, not the raw table.

Mixed workload — streaming inserts and analytical queries on the same engine, no ETL step, no separate OLAP warehouse. One port, one connection string, one database.

Compression — the same data that would be ~50KB in Postgres sits at 14KB in ClickHouse. At scale, this compounds: 100M rows that would be ~40GB in row storage compress to ~4GB.

SEO Intelligence Layer

The 18 snapshot files contain richer SEO data beyond raw traffic numbers: health scores (composite 0–100 across traffic, ranking, indexing, content, CWV, conversion), AIO impact (estimated clicks lost to Google's AI Overviews), and ranking movement (which keywords improved or declined week-over-week).

These are ingested as fields on WeeklySnapshot alongside the standard metrics. The same ClickHouse table powers both the traffic trend charts and the SEO intelligence KPIs — no separate system.

AIO (AI Overview) tracking is particularly meaningful in 2026: when a query triggers an AI Overview, organic CTR drops sharply even for position-1 results. The aioLostClicks field tracks estimated weekly click loss, which is a leading indicator of GEO (Generative Engine Optimization) pressure.

Schema Normalization Challenge

The 18 JSON snapshots evolved through 6+ different structural variants as the collection scripts improved. A simple d["plausible"]["visitors"] access would fail on half the files.

The backfill script uses a priority-chain extraction pattern for each metric — it tries field paths in order of likelihood and falls back gracefully:

# visitors: try 5 different locations across schema versions
for path in [
  lambda p: p.get("visitors"),
  lambda p: p.get("agg_w1", {}).get("visitors"),
  lambda p: p.get("aggregate", {}).get("visitors"),
  lambda p: p.get("aggregate_m1", {}).get("visitors"),
  lambda p: p.get("visitors_w1"),
]:
  v = path(plausible_block)
  if v is not None:
    return int(v)
This is a real-world data engineering pattern. Production data sources evolve. Robust extraction chains mean new schema versions are additive — you add a new fallback without breaking existing parsers.

GitHub Webhook Adapter

GitHub puts the event type (star, fork, issues) in the HTTP header (X-GitHub-Event), not the JSON body. MooseStack's ingest endpoint only sees the JSON body.

Solution: a thin Flask proxy on port 3001 extracts the header and injects it into the payload before forwarding to MooseStack:

# webhook_adapter.py — 20 lines, solves the impedance mismatch
@app.route('/webhook', methods=['POST'])
def relay():
  event_type = request.headers.get('X-GitHub-Event', 'unknown')
  delivery_id = request.headers.get('X-GitHub-Delivery', str(uuid4()))
  body = request.get_json(force=True) or {}
  actor = (body.get('sender') or {}).get('login', 'unknown')

  payload = {
    'deliveryId': delivery_id,
    'timestamp': datetime.now(timezone.utc).isoformat(),
    'eventType': event_type,
    'repo': (body.get('repository') or {}).get('full_name', ''),
    'actor': actor,
    'action': body.get('action', 'created'),
    'rawPayload': json.dumps(body),
  }
  # Forward to MooseStack ingest endpoint
  requests.post('http://localhost:4000/ingest/GithubEvent', json=payload)
